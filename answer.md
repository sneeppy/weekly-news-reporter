# Решение тестового задания

## 1) Какие инструменты эффективнее всего использовать

### Источники новостей (как собирать)

- **RSS/Atom**: быстрый и надёжный способ для тех источников, где он есть (многие медиа/блоги/порталы).
- **Google News RSS по запросам**: быстро расширяет охват (но качество нужно фильтровать).
- **Платные/условно‑платные API** (если нужен масштаб/обогащение):
  - **NewsAPI / MediaStack**: агрегаторы (лимиты, лицензии).
  - **GDELT**: большой объём, удобно для аналитики/трендов.
- **Веб‑скрейпинг** (если RSS/API нет): Scrapy + Playwright (для JS‑сайтов), но дороже по поддержке.

### Хранилище и обработка

- **SQLite** (прототип/малые объёмы) или **PostgreSQL** (прод, несколько источников/пользователей).
- **Дедупликация**:
  - по URL (первичная),
  - по нормализованному заголовку,
  - по похожести текста (SimHash/MinHash) для “перепечаток”.
- **Классификация по темам**:
  - для старта: **правила/ключевые слова** (дешево, прозрачно),
  - дальше: ML‑классификатор / LLM‑классификация (с проверками).
- **Суммаризация**:
  - если критично исключать выдумки — **экстрактивная** (цитаты/фразы из источника),
  - если нужен “аналитический текст” — LLM, но строго “grounded” (см. раздел 3).

### Оркестрация, доставка, наблюдаемость

- **Планировщик**: cron/Windows Task Scheduler для простого кейса; Prefect/Airflow — если пайплайн сложный.
- **Доставка**: email/Telegram/Slack + публикация в Notion/Confluence.
- **Мониторинг**: логирование, метрики (сколько новостей, ошибки парсинга, источники‑аутлаеры), алерты.

## 2) Как настроить автоматизацию наиболее эффективным образом

### Рекомендуемая архитектура (поток данных)

1) **Ингест** (каждые 1–6 часов)
   - загрузка RSS/API/страниц,
   - нормализация полей: `title/link/date/source/summary`.
2) **Очистка** (pipeline)
   - вычищение HTML,
   - унификация дат/таймзон,
   - фильтр “мусора” (слишком короткие/подозрительные элементы).
3) **Дедуп** (pipeline)
   - `link` unique,
   - затем fuzzy‑дедуп по заголовку/содержимому.
4) **Тематизация**
   - keyword‑матчинг (быстро и прозрачно),
   - по мере роста: модель классификации.
5) **Недельный отчёт** (1 раз в неделю)
   - “срез”: последние 7 дней,
   - аналитика: распределение по темам, источникам, дням; топ‑термины,
   - подборка “топ N” по каждой теме,
   - итоговый Markdown/PDF + отправка.

### Практические детали (чтобы работало месяцами)

- **Конфигурируемость**: темы/ключевые слова/ленты — в конфиге (не в коде).
- **Повторяемость**: хранить “сырые” данные (минимум `title/link/date/summary`) + версию отчёта.
- **Идемпотентность**: повторный запуск не должен дублировать записи.
- **Тайм‑ауты и ретраи**: сетевые ошибки не должны валить весь отчёт.
- **Качество**: добавлять “белые списки” источников, исключения доменов, стоп‑слова.

## 3) Как полностью исключить галлюцинации

Ключевой момент: **«полностью» гарантировать отсутствие галлюцинаций можно только если отчёт формируется без генеративных утверждений** (экстрактивно или из детерминированной статистики).

### Вариант A (100% безопасный): без LLM

- отчёт содержит:
  - список новостей с ссылками,
  - краткое описание **строго из RSS/лида** или первых предложений (после очистки),
  - аналитику по счётчикам (сколько новостей, топ‑источники, топ‑слова).
- никаких “фактов” сверх того, что уже дано источниками.

### Вариант B (LLM, но “grounded” + жёсткие правила)

Если нужен “аналитический текст”, то:

- **Давать модели только подтверждённый контекст**: заголовок + короткий сниппет + ссылка + (опционально) несколько цитат из страницы.
- **Требовать цитирования**: каждое утверждение должно иметь ссылку/ID источника.
- **Запрет на числа/факты без цитаты**: модель должна писать “не указано в источниках”.
- **Структурированный вывод (JSON schema)** + валидация.
- **Пост‑проверка**:
  - автоматический чек: все ссылки в отчёте существуют,
  - утверждения с числами/датами должны быть “привязаны” к источнику,
  - если не проходит — модель перегенерирует с режимом “abstain”.

Но даже так “абсолютный ноль” рисков в генеративном тексте не доказуем; поэтому для требования “полностью исключить” корректный ответ — **делать отчёт экстрактивным/ссылочным**, а “аналитику” ограничить статистикой и перефразированием только в пределах цитируемого.

## 4) Прототип и промт

В `prototype/` лежит рабочий скрипт, который:

- собирает новости из RSS/Atom,
- сохраняет их в SQLite (идемпотентно),
- строит недельный отчёт в Markdown,
- добавляет аналитику (распределение по темам/дням/источникам, топ‑термины),
- не использует LLM → **галлюцинаций нет** (все строки привязаны к источнику).

### Промт для LLM‑версии (если нужно “красиво написать”, но без выдумок)

**System:**
Ты пишешь недельный дайджест новостей. Ты НЕ имеешь права добавлять факты, которых нет в входных данных. 
Если информации недостаточно — так и скажи. Любое утверждение должно ссылаться на источники по их id.
Запрещено придумывать цифры, даты, имена, причины, последствия.

**User:**
Данные за неделю (список источников). Для каждого источника есть: id, title, url, date, snippet.
Сформируй отчёт:
- В начале: краткая сводка (только обобщение по темам и количествам, которые я дал).
- Затем по темам: 5–10 пунктов. Каждый пункт: одно предложение + (id источника в квадратных скобках) + ссылка.
Если тема пуста — напиши “нет релевантных новостей”.
